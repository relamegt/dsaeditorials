# Signal Handling in C++

## Introduction

Signals are asynchronous notifications sent by the operating system (or generated by the process itself) to inform a running program that a specific event has occurred. These events can represent system errors (like a division by zero or access violation), user commands (such as Ctrl+C to terminate a process), or termination requests. 

When a signal is sent to a process, the operating system interrupts the normal execution flow to trigger either a default system action or a user-defined handler.

### Key features of signals:

- Notify processes of system events or hardware exceptions.
- Have default behaviors (usually program termination) defined by the OS.
- Can be intercepted and managed using custom callback functions called **signal handlers**.
- Declared and imported via the `<csignal>` header.

## List of Common C++ Signals

| Signal Name | Description | Default Action |
| --- | --- | --- |
| `SIGABRT` | Abnormal program termination (e.g., triggered by `std::abort()`) | Terminate program |
| `SIGFPE` | Erroneous arithmetic operation (e.g., divide by zero, overflow) | Terminate program |
| `SIGILL` | Invalid hardware instruction encountered | Terminate program |
| `SIGINT` | Interactive user interrupt (sent when pressing Ctrl+C) | Terminate program |
| `SIGSEGV` | Invalid memory access (segmentation fault) | Terminate program |
| `SIGTERM` | Termination request sent to the program | Terminate program |
| `SIGUSR1` | User-defined signal 1 (available for custom application logic) | Terminate program |
| `SIGKILL` | Forces the OS to terminate the process immediately | Terminate program (cannot be caught) |
| `SIGSTOP` | Forces the OS to pause execution | Stop program (cannot be caught) |

## Signal Handling in C++

Signal handling is the process of overriding a signal's default behavior with a custom function. In C++, this is achieved using the `std::signal()` function from the `<csignal>` library.

**Syntax:**

```Cpp
signal(signal_type, signal_handler);
```

- **`signal_type`:** The constant representing the signal (e.g., `SIGINT`).
- **`signal_handler`:** A pointer to a function matching the signature `void handler(int)`. Alternatively, you can pass constants like `SIG_DFL` (restore default behavior) or `SIG_IGN` (ignore the signal).

### Example: Catching the Interrupt Signal (SIGINT)

```Cpp
#include <iostream>
#include <csignal>
#include <unistd.h>
using namespace std;

// Custom signal handler function
void handleUserInterrupt(int sig) {
    cout << "\nGraceful shutdown initiated (Signal: " << sig << ")" << endl;
    
    // Perform cleanup operations here
    cout << "Releasing resources..." << endl;
    
    // Terminate program safely
    exit(sig);
}

int main() {
    // Override SIGINT (Ctrl+C) with the custom handler
    signal(SIGINT, handleUserInterrupt);

    cout << "Background worker active. Press Ctrl+C to terminate..." << endl;

    // Loop simulating continuous background execution
    while (true) {
        cout << "Active service cycle running..." << endl;
        sleep(1); // Sleep for 1 second
    }

    return 0;
}
```

### Output
Background worker active. Press Ctrl+C to terminate...
Active service cycle running...
Active service cycle running...
Active service cycle running...
^C
Graceful shutdown initiated (Signal: 2)
Releasing resources...

When the user enters Ctrl+C, the OS sends `SIGINT` to the process. Instead of terminating instantly, the program runs `handleUserInterrupt()`, performs cleanup, and exits gracefully with status `2` (the value of `SIGINT`).

## Raising a Signal Manually

A process can send a signal to itself programmatically. This is useful for testing signal handlers or triggering error routines.

### The `raise()` Function

The `raise()` function sends a specified signal to the current process.

**Syntax:**

```Cpp
raise(signal_type);
```

```Cpp
#include <iostream>
#include <csignal>
using namespace std;

void handleTermination(int sig) {
    cout << "Termination request processed (Signal: " << sig << ")" << endl;
    exit(sig);
}

int main() {
    // Bind SIGTERM to the handler
    signal(SIGTERM, handleTermination);

    cout << "Raising SIGTERM internally..." << endl;
    
    // Manually trigger the signal
    raise(SIGTERM);

    cout << "This line will not execute." << endl;
    return 0;
}
```

### Output
Raising SIGTERM internally...
Termination request processed (Signal: 15)

## Sending Signals to Other Processes

In POSIX environments, you can send signals between different running processes.

### The `kill()` Function

The `kill()` function sends a signal to a specific process or process group identified by its Process ID (PID).

**Syntax:**

```Cpp
kill(pid, signal_type);
```

```Cpp
#include <iostream>
#include <csignal>
#include <unistd.h>
using namespace std;

void handleUserSignal(int sig) {
    cout << "Custom user signal processed (Signal: " << sig << ")" << endl;
}

int main() {
    // Bind the custom user-defined signal SIGUSR1
    signal(SIGUSR1, handleUserSignal);

    // Retrieve the Process ID (PID) of the current process
    pid_t pid = getpid();
    cout << "Current Process ID (PID): " << pid << endl;

    // Send SIGUSR1 to our own process ID
    cout << "Sending SIGUSR1 to PID: " << pid << endl;
    kill(pid, SIGUSR1);

    cout << "Execution completed successfully." << endl;
    return 0;
}
```

### Output
Current Process ID (PID): 4510
Sending SIGUSR1 to PID: 4510
Custom user signal processed (Signal: 10)
Execution completed successfully.

In this program, we use `getpid()` to find the running process ID, then use `kill()` to send the user-defined signal `SIGUSR1`. The handler processes it, and execution continues because the handler did not call `exit()`.

> **Note:** Inside signal handlers, only **async-signal-safe** functions should be called. Functions like `std::cout` or `malloc()` are unsafe because they are not reentrant and use global locks internally; calling them inside an interrupt can cause deadlocks if the main thread was interrupted while holding those same locks. In production systems, handlers should only set volatile atomic flags for the main loop to read.

## Summary

Signal handling in C++ provides a way for processes to respond to operating system notifications and hardware interrupts. The `std::signal()` function registers a custom handler for a given signal. Signals can be generated automatically by hardware events or user input, raised internally using `raise()`, or sent between processes using the `kill()` function. Using custom handlers allows applications to intercept termination requests and perform cleanup before exiting, though safety guidelines must be followed inside the handler code.
